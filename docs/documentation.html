MCP Tools

Six tools exposed via FastMCP. Each tool encapsulates one step of the validation pipeline.

step1_classify_question

Step 1 gpt-4o-mini

Gate check: determines if the question relates to SFC retail data, SFC non-retail, or general knowledge. Non-retail questions are answered directly and exit the pipeline.

Input: question (string)   Output: classification, reasoning, answer

Technical Details ▶

Uses a single LLM call with a classification system prompt. Returns JSON with classification (sfc_retail|sfc_non_retail|general), reasoning, and answer (null for sfc_retail).

Key disambiguation: "What is a stuffed animal?" → general (definitional), vs "How many stuffed animals sold?" → sfc_retail (data query).

System prompt excerpt:
Classify each question into one of three categories:
- "sfc_retail": Questions requesting retail data, analytics, or metrics
  about Sevilla FC merchandise.
- "sfc_non_retail": Questions about Sevilla FC but NOT about retail.
- "general": Questions asking for definitions, explanations, or general
  knowledge — even if the subject is an SFC product.

step2_validate_retail_domain

Step 2 gpt-4o-mini gpt-4o

Two-phase validation: Phase A extracts dimensions (concept, metric, date, location) via LLM. Phase B validates each dimension — concept via ontology + LLM, date via Python, location via LLM, metric via Python.

Input: question   Output: is_valid, concept, date_range, location, metric, suggestion

Technical Details ▶

Concept validation (3-tier cascade):

Tier 1: Search ontology → LLM confirms semantic match (gpt-4o).
Tier 2: No ontology match → search DuckDB master_articles directly.
Tier 3: No match anywhere → INVALID.

Date validation (deterministic Python): Parses month names (EN/ES), full date patterns, validates range 2012–today.

Location validation (LLM): Earth-based = VALID, fictional/space = INVALID.

Metric validation (deterministic Python): Rejects subjective metrics (happiness, feelings) and privacy-sensitive requests (individual customer/employee data).

Extraction prompt:
Extract structured fields from the user's retail question:
{
  "concept": "product or category, or null",
  "metric": "measurable metric, or null",
  "date_range": "time period, or null",
  "location": "location mentioned, or null"
}

step3_map_to_ontology

Step 3 gpt-4o-mini (disambiguation)

Maps product concepts to the 3-level hierarchy. Uses in-memory index for search (exact → substring → fuzzy), with LLM only when multiple matches need disambiguation.

Input: concept, question   Output: matched_nodes, granularity, disambiguation_needed

Technical Details ▶

When concept is null (e.g., "total billing in 2025"), query applies to all items — no product filter.

Disambiguation prompt asks LLM to select the most relevant match(es) from a numbered list when duplicate product names exist in different hierarchy paths.

Granularity output: family (level 1), subfamily (level 2), product_category (level 3), or product_type (level 4). Falls back to DuckDB item search if ontology has no results.

step4_tlk_lookup

Step 4 o4-mini

Template matching + SQL generation. Normalizes the question (replaces concrete values with placeholders), compares against 25 TLK templates, and assigns confidence mechanically.

Input: question, ontology_context, date_range   Output: confidence, sql, source, explanation

Technical Details ▶

3 code paths:

1. ACCURATE (exact match): Fill template placeholders with actual values from ontology/date context. No LLM needed for SQL.

2. MEDIUM (partial match): LLM adapts the closest template's SQL with DDL + business rules context.

3. LOW (no match): LLM generates SQL from scratch using DDL, ontology, and business rules.

Uses o4-mini (reasoning model) for all LLM calls in this step — chosen for its ability to reason about complex SQL queries.

step5_execute_query

Step 5 gpt-4o-mini

Executes the generated SQL against local Parquet files via DuckDB, then summarizes results in natural language. LOW confidence queries get a warning banner.

Input: sql, confidence, question   Output: results_csv, row_count, summary, warning

Technical Details ▶

DuckDB connection is cached in-memory. All Parquet files from data/parquets/ are registered as views under both retail and retail2 schemas at startup.

Summary prompt provides the first 50 rows of CSV, the original question, confidence level, and row count. LLM responds in 2-3 sentences with key numbers in EUR.

Summary system prompt:
You are a retail analytics assistant for Sevilla FC.
Summarize query results clearly and concisely in 2-3 sentences.
Use EUR for monetary values.

Product Ontology

Sevilla FC's SAP master data contains ~29,000 SKUs with inconsistent naming spanning a decade of seasons. The ontology creates a three-level product hierarchy (Family > Subfamily > Product Type) so the LLM can work with a compact, structured taxonomy instead of raw product names.

Why It Exists

Without structured taxonomy, it's impossible to ask analytical questions like "how did Textile-Competition revenue evolve across seasons?" The ontology maps 22,413 active SKUs (items that appear in actual transactions) into 10 families, 21 subfamilies, and 3,284 product types.

Ontology Search Deep Dive

The ontology search is one of the most complex parts of the pipeline. It is used by two different MCP tools (Step 2 and Step 3) for different purposes, and involves translation, multi-phase matching, disambiguation, and a DuckDB fallback. This section explains every detail.

Which Tools Use the Ontology Search?

step2_validate_retail_domain

Step 2 gpt-4o

Purpose: Validates that the user's product concept actually exists in Sevilla FC's catalog. The search finds candidate matches, then an LLM (gpt-4o) confirms semantic equivalence.

Source: validate.py:_validate_concept()

Uses: search_ontology() + search_items_duckdb()

Output: is_valid: true/false — a binary gate (proceed or reject).

step3_map_to_ontology

Step 3 gpt-4o-mini

Purpose: Maps the validated concept to specific ontology nodes so the next step knows which SKUs to query. Includes disambiguation when the same product name exists in multiple hierarchy paths.

Source: ontology_map.py:map_to_ontology()

Uses: search_with_fallback() + find_duplicates() + _disambiguate()

Output: matched_nodes[] with paths, levels, SKU counts + granularity.

Complete Search Architecture

The search system is built from four modules that work together:

Phase 0: Translation Expansion

Before any searching begins, the input term is expanded into a list of search terms using a bidirectional EN↔ES dictionary of ~110 retail/football terms.

Source: translations.py:get_all_translations()

How it works

  • The dictionary maps English retail terms to Spanish: "shirt" → "camiseta", "boots" → "botas", "training" → "entrenamiento"
  • A reverse map is auto-generated: "camiseta" → "shirt"
  • Multi-target synonyms expand further: "shorts" → ["pantalon corto", "calzona", "short", "bermuda"]
  • The result is always a list containing the original term + all translations
Example: User says "How many shorts sold?" → concept = "shorts" ↓ get_all_translations("shorts") returns: ["shorts", "pantalon corto", "calzona", "short", "bermuda"] ↓ All 5 terms are searched against every node in the index.

Phases 1-3: The Search Cascade

Source: search.py:search_ontology() — iterates over all ~4,700 index nodes and classifies each match into one of three priority buckets.

Phase 1: Exact Match (highest priority)

For each expanded search term, checks:

  • Does the node's canonical name (product_type or category or subfamily or family) equal the search term exactly? (case-insensitive)
  • Is the search term one of the node's path parts? e.g. searching "entrenamiento" matches node TEXTIL/Entrenamiento/CAMISETA because "entrenamiento" is a path part.

If either matches, the node is added to the exact bucket and skipped for later phases.

Phase 2: Substring Match

For nodes not matched by Phase 1, checks if any expanded search term appears as a substring within:

  • The node's search_text — a pre-built, lowercased string containing all path parts + all their translations + sample item names (for level 4 nodes)
  • The node's canonical name

This catches partial matches like searching "bota" matching a node named "BOTA DE FUTBOL".

Phase 3: Fuzzy Match (lowest priority)

For remaining unmatched nodes, uses Python's difflib.SequenceMatcher:

  • Compares each search term against the node name AND against each individual path part
  • Takes the best score across all comparisons
  • Threshold: 0.6 (60% similarity) — configured via FUZZY_MATCH_THRESHOLD
  • This catches misspellings and morphological variations (e.g. "camisetas" → "CAMISETA" scores ~0.89)

Result Combination

The three buckets are combined in priority order: exact > substring > fuzzy (sorted by score descending). Then:

  • Deduplicated by path — if the same node was matched by multiple search terms, only the first (highest priority) occurrence is kept
  • Capped at 20 results (configurable via max_results)

DuckDB Fallback

Source: search.py:search_items_duckdb() — only triggered when the ontology search returns zero results.

How it works

Queries the master_articles_oitm.parquet file directly via DuckDB:

  • Tokenizes multi-word terms: "nike air" becomes two tokens
  • Each token must appear in item_name (AND logic): item_name ILIKE '%nike%' AND item_name ILIKE '%air%'
  • Also checks item_code ILIKE for code-based searches
  • Filters out (NO USAR) test items and unused item groups (itms_grp_cod = 100)
  • Returns up to 20 ItemMatch objects with item_code + item_name

This catches products that exist in the raw database but weren't captured by the ontology hierarchy (e.g., very new items or unusual product names).

Disambiguation

Source: ontology_map.py:_disambiguate() — triggered in Step 3 only when a product name appears in multiple hierarchy paths.

When does it trigger?

After the search returns nodes, Step 3 takes the best match's name and calls find_duplicates(name). If the same name exists in more than one path, disambiguation is needed.

Example: User asks "How many CAMISETA sold in training?" ↓ Search returns nodes for "CAMISETA" — but CAMISETA exists in multiple paths: ↓ 1. TEXTIL/Competicion/CAMISETA (784 SKUs)
2. TEXTIL/Entrenamiento/CAMISETA (456 SKUs)
3. TEXTIL/General/CAMISETA (2,278 SKUs)
4. TEXTIL/Casual/CAMISETA (39 SKUs)
5. COMPLEMENTO/Competicion/CAMISETA (5 SKUs) ↓ LLM (gpt-4o-mini) receives all 5 options + the original question. Because the user said "training", it selects index 2 (Entrenamiento). ↓ Output: {"selected_indices": [2]} → only TEXTIL/Entrenamiento/CAMISETA is returned.

Disambiguation prompt

The prompt sent to gpt-4o-mini includes:

  • The original user question for context
  • A numbered list of all candidate paths with SKU counts
  • Instruction: "Return JSON with selected_indices (1-based). If unsure, include all. Prefer the most specific match."

If the LLM call fails, all candidates are returned (safe fallback).

Question: "Total billing for shirts in December 2025" ↓ Step 2 extracts: concept = "shirts", date = "December 2025" ↓ Step 2 validates concept:
1. search_ontology("shirts", max_results=10)
2. Translation: "shirts" → ["shirts", "camiseta", "polo"]
3. Exact match: "CAMISETA" nodes found (784 + 456 + 2,278 + ... SKUs)
4. LLM (gpt-4o) confirms: concept "shirts" corresponds to CAMISETA matches → VALID ↓ Step 3 maps to ontology:
1. search_with_fallback("shirts")
2. Translation: "shirts" → ["shirts", "camiseta", "polo"]
3. Multiple CAMISETA nodes found across Competicion, Entrenamiento, General, Casual, etc.
4. find_duplicates("CAMISETA") → 5 paths → disambiguation needed
5. LLM (gpt-4o-mini): no specific subfamily in question → returns all CAMISETA paths
6. Output: granularity = "product_category", multiple matched nodes ↓ Step 4 uses the nodes to build SQL with appropriate ILIKE filters for all CAMISETA variants.

System Prompts

1. Classification System Prompt (classify.py) ▶
You are a question classifier for Sevilla FC's retail analytics system.

Classify each question into one of three categories:
- "sfc_retail": Questions requesting retail data, analytics, or metrics about
  Sevilla FC merchandise — sales numbers, billing, inventory counts, stock
  levels, revenue, etc. The question must ask for data, not a definition.
- "sfc_non_retail": Questions about Sevilla FC but NOT about retail (match
  results, player transfers, stadium capacity, club history, etc.)
- "general": Questions asking for definitions, explanations, or general
  knowledge — even if the subject is an SFC product. Also includes topics
  unrelated to Sevilla FC entirely.

Disambiguation examples:
- "What is a stuffed animal?" → general (definitional, not a data query)
- "How many stuffed animals sold?" → sfc_retail (data query)
- "What is a Nintendo Switch?" → general (definitional)
- "How many Nintendo Switches have we sold?" → sfc_retail (data query)

Respond in JSON with:
{
  "classification": "sfc_retail" | "sfc_non_retail" | "general",
  "reasoning": "brief explanation",

Business Rules

Store Filter (6 Retail Stores)

ALL queries on inv1/rin1 must filter by these 6 retail cost centers:

ocr_code IN ('COM0201', 'COM0202', 'COM0206', 'COM0207', 'COM0208', 'COM0209')
COM0201Tienda EstadioCOM0207Tienda Eventos
COM0202Tienda OnlineCOM0208Tienda Distribucion
COM0206Tienda CentroCOM0209Tienda Airesur

Net Billing Formula

Revenue is calculated as invoices minus returns:

SUM(inv1.line_total) - SUM(rin1.line_total)

Similarly for quantity: SUM(inv1.quantity) - SUM(rin1.quantity)

Item Exclusion Filter

Test/deprecated items must always be excluded:

item_name NOT LIKE '%(NO USAR)%'

Product matching uses case-insensitive: item_name ILIKE '%term%'
